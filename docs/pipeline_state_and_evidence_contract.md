# Pipeline State and Evidence Contract

## Scope

This document defines the single-host persistence model for a pipeline run. SQLite is the query and transition authority; filesystem artifacts provide immutable or append-only evidence addressed by database records.

## Database location and migrations

Each host declares a relative SQLite path in `pipeline.yaml`, normally `state/pipeline.sqlite`. The database must be created only by an explicit preflight/initialization path, use WAL journal mode when supported, and run versioned, transactional migrations before work selection. A runner refuses to process work if a migration cannot complete or the database schema is newer than the runtime supports.

## Logical tables

| Table | Required purpose |
| --- | --- |
| `schema_migrations` | Applied migration identity, timestamp, and runtime version. |
| `pipeline_revisions` | Pipeline ID, schema version, normalized definition hash, source revision, and creation time. |
| `entities` | Stable entity ID, discovery key, canonical source path, content type, and discovery metadata. |
| `entity_revisions` | Immutable source hash/snapshot, revision number, pipeline revision, disposition, and current state. |
| `runs` | Run ID, command arguments, host/runtime/config fingerprints, start/end time, status, and aggregate counters. |
| `run_entities` | Membership and per-run scheduling outcome for an entity revision. |
| `attempts` | Attempt number, run, stage, lease owner, start/end time, state, retry classification, and error summary. |
| `state_transitions` | Append-only from/to state, timestamp, actor, reason code, and evidence references. |
| `validator_results` | Validator ID/version, pass/fail/skipped, stable code, explanation, metrics, and evidence reference. |
| `review_results` | Reviewer contract/version, verdict, confidence, violations, action, and thread reference. |
| `artifacts` | Artifact kind, immutable content hash, path, byte size, retention status, and relationship to entity revision. |
| `thread_captures` | Call/stage/attempt identity, capture path/hash, provider metadata, timing/usage availability, and redaction flags. |
| `leases` | Current owner, acquired/expires timestamps, heartbeat, and safe-release token. |
| `cohorts` | Report/run identity, deterministic grouping key, count, and representative evidence selection. |
| `remediation_proposals` | Advisory proposal schema/version, cohort reference, evidence links, and review disposition. |

Foreign-key constraints and transition guards prevent dangling evidence and invalid state changes. JSON payloads are allowed only for bounded structured metadata; fields that determine querying, safety, or transitions remain normalized columns.

## Entity identity and revisions

Discovery computes an entity ID from a declared stable discovery key, not solely a mutable filesystem path. For a Markdown file, the default key may be a normalized repository-relative path; a project can supply a content-domain key where a rename must retain identity.

At staging, the runtime records the source byte hash and creates an entity revision. If the source hash, configured prompt contract, or pipeline revision differs from a prior accepted/rejected revision, a new revision is required. The old revision remains auditable. If all idempotency inputs match an accepted terminal outcome, a resumed run records reuse rather than calls the model again.

The idempotency key is the hash of entity revision ID, pipeline revision hash, stage contract version, normalized prompt inputs, and requested output contract. A stage never uses a previous output with a mismatched key.

## State transition rules

Only the transaction that owns a valid lease may advance a non-terminal entity revision. Required transitions are recorded atomically with the corresponding attempt/evidence reference.

| State | Allowed next states |
| --- | --- |
| `discovered` | `staged`, `quarantined` |
| `staged` | `leased`, `quarantined` |
| `leased` | `worker_completed`, `retry_eligible`, `quarantined` |
| `worker_completed` | `deterministic_validated`, `validation_failed`, `retry_eligible`, `quarantined` |
| `deterministic_validated` | `semantic_reviewed`, `accepted`, `repair_eligible`, `quarantined` |
| `semantic_reviewed` | `accepted`, `repair_eligible`, `retry_eligible`, `quarantined` |
| `repair_eligible` or `retry_eligible` | `leased`, `quarantined` |
| `accepted` | `promoted`, `quarantined` only if promotion safety fails before any source change |
| `promoted`, `quarantined` | terminal |

A lease has a random owner token, a finite expiry, and an optional heartbeat. A new runner may reclaim it only after expiry and records a `lease_expired` transition. It never assumes an expired worker left no candidate artifact; it creates a new attempt and leaves prior evidence untouched.

## Artifact policy

Artifact paths are generated by the runtime from safe IDs and cannot be influenced by model output. Each written artifact is first created in a temporary file under its declared root, fsynced where supported, content-hashed, and atomically renamed into place.

| Artifact kind | Default root | Mutability |
| --- | --- | --- |
| Source snapshot | `artifacts/source/` | Immutable per entity revision |
| Staged candidate | `artifacts/staged/` | Immutable per attempt |
| Validator evidence | `artifacts/evidence/` | Immutable per validation invocation |
| Accepted candidate | `artifacts/accepted/` | Immutable provenance copy |
| Quarantine bundle | `artifacts/quarantined/` | Append-only evidence view |
| Promotion backup | configured rollback root | Immutable until retention expiry |

Promotion copies or atomically swaps only an accepted candidate to a configured target after a promotion record and rollback copy are durable. The original source is not modified until this point. If promotion fails, the revision becomes `quarantined` with `promotion_failed`; the source is unchanged. Rollback uses the recorded backup and is an explicit operator action, never a model recommendation.

## Thread-capture and evidence integrity

Every inference attempt creates a canonical JSON capture in the declared thread root before a stage result is committed. Its byte hash is stored in `thread_captures`; state transitions refer to that row. Captures contain a redaction manifest, not secrets. A missing or corrupt required capture makes the attempt ineligible for acceptance and creates an operational error requiring retry or quarantine.

Validator and reviewer evidence follows the same content-addressed artifact pattern. Reports link by database ID and content hash, not only by a relative filename, so artifacts may later be relocated to cold storage without corrupting history.

## Retention and deletion

The definition specifies retention windows by artifact kind and whether full prompt/response bodies are stored, redacted, hashed-only, or disabled. Runtime cleanup may delete only non-required, expired evidence after confirming no nonterminal entity revision or pending report references it. Cleanup produces an audit entry. Credentials are never stored in the state database or artifact tree.

## Concurrency and crash safety

The first implementation is single-host: one process holds a runner lock for a pipeline state database. The lock includes process ID, run ID, creation time, and host identifier; stale lock recovery requires a conservative age threshold and an audit record. SQLite transactions use `BEGIN IMMEDIATE` or equivalent short write transactions; model calls occur outside a database write transaction. Crash recovery always favors a duplicate-safe retry over assuming success.
